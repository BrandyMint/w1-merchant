Поставщики услуг

Получение доступных поставщиков услуг

GET /OpenApi/providers?page={page}&itemsPerPage={itemsPerPage}

    &providerGroupId={providerGroupId}&locationId={locationId}

    &searchString={searchString}&isFavourite=true

Обязательные параметры строки запроса:

  • page — номер страницы, начиная с 1
  • itemsPerPage — количество элементов на странице

Необязательные параметры строки запроса:

  • providerGroupId — идентификатор группы поставщиков услуг.
  • locationId — идентификатор местоположения.
  • searchString — строка для поиска по описанию поставщика услуг.
  • isFavourite=true — выбираются поставщики услуг из Избранного.

Пример ответа:

{

   "Providers":[ // поставщики услуг

      {

         // Идентификатор поставщика услуг

         "ProviderId":"abc",

        // Идентификатор пользователя, соответствующий поставщику

         "UserId":100000000000,

         // Название поставщика услуг

         "Title":"ABC Usluga",

         // Описание поставщика услуг

         "Description":"ABC Described",

         // URL поставщика услуг

         "Url":"http://abc.com/",

         // Идентификатор валюты, согласно ISO 4217

         "CurrencyId":643,

         // Входит ли поставщик услуг в Избранное

         "IsFavourite":true,

         // Флаг, показывающий, что поставщик услуг включен

         "IsActive":true,

         // Ссылка на логотип

         "LogoUrl":"http://domain.com/picture.jpg",

         // Описание комиссии

         "Commission":

         {

            "Rate":2.0,

            "Cost":0.0,

            "Min":0.0,

            "Max":0.0

         }

      },

      ... // остальные поставщики услуг на запрошенной странице

   ],

   // Общее число поставщиков услуг,

   // удовлетворяющих заданным критериям

   "TotalCount":524

Управление Избранным поставщиков услуг

PUT /OpenApi/providers/{providerId}/?isFavourite={isFavourite}

Тело запроса и ответа пустое.

Если в isFavourite указано  true, поставщик услуг будет добавлен в Избранное.

Если в isFavourite указано  false, поставщик услуг будет исключен из Избранного.

Платежи

Инициализация платежа

POST /OpenApi/payments

Пример тела запроса:

{

     // Идентификатор поставщика услуг

     "ProviderId":"mts",

     // Уникальный в пределах пользователя идентификатор платежа, в системе учета приложения. Необязательный параметр.

     "ExternalId":"ХХХХХХХ"

}

Сервис поддерживает два способа идентификации платежей: каждый платеж имеет уникальный идентификатор, назначенный сервисом (обозначается PaymentId), а также может иметь уникальный идентификатор, назначенный пользователем (обозначается ExternalId).

В случае, когда пользователь назначает платежам идентификаторы, нельзя назначать один и тот же идентификатор различным платежам. Не рекомендуется использовать длинные (более 40 символов) строки в качестве идентификаторов платежей.

Пользователь может не назначать создаваемым платежам идентификаторы. В этом случае поле ExternalId не передается.

Сервис возвращает описание формы, которую необходимо заполнить. Платеж принимается к обработке после заполнения всех форм. Пример ответа:

{

   // Идентификатор платежа, назначенный сервисом

   "PaymentId":4444444,

   // Идентификатор поставщика услуг

   "ProviderId":"mts",

   // Состояние платежа

   "State":{

      "StateId":"Created", // Возможные состояния

      "Description":""

   },

   // Номер шага

   "Step":1,

   // Общее количество шагов

   "StepCount":3,

   // Идентификатор основного поля, требующего заполнения

   "MasterFieldId":"phone",

   // Описание формы, которую необходимо заполнить

   "Form":{

      // Идентификатор формы платежа

      "FormId":"Index",

      // Название формы платежа

      "Title":"МТС",

      // Описание формы платежа

      "Description":"Для оплаты заполните платежные поля",

      // Поля, которые необходимо заполнить

      "Fields":[

         {

            // Тип поля

            "FieldType":"Scalar",

            // Значение поля по умолчанию

            "DefaultValue":"9128449877",

            // Пример значения поля

            "Example":"9128449877",

            // Регулярное выражение, которому должно

            // удовлетворять значение поля

            "RegEx":"^\\d{10}$",

            // Минимальная длина значения

            "MinLength":10,

            // Максимальная длина значения

            "MaxLength":10,

             // Идентификатор поля

            "FieldId":"phone",

            // Название поля

            "Title":"Номер телефона",

            // Описание поля

            "Description":"Введите 10 цифр номера телефона",

            // Позиция поля в форме

            "TabOrder":1,

            // Флаг, указывающий, является ли поле обязательным

            "IsRequired":true,

            // Маска поля для ввода

            "InputMask":"(999) 000-00-00;0;-",

         },

         {

            "FieldType":"Scalar",

            "DefaultValue":"10",

            "Example":"",

            "RegEx":"^\\d+(\\.\\d{1,4})?$",

            "MinLength":1,

            "MaxLength":10,

            "FieldId":"Amount",

            "Title":"Сумма (руб.)",

            "Description":"Введите сумму платежа",

            "TabOrder":2,

            "IsRequired":true,

         }

      ]

   }

}

Поля формы, которые необходимо заполнить для проведения платежа, разделяются на три типа:

  • Scalar (скалярные) —  поля, значения которых можно описать одним числом или строкой (номер договора, номер счета, телефона, примечание, ФИО и т.п.);
  • List (списковые) — поля, значения которых необходимо выбрать из предопределённого списка. В качестве компонента пользовательского интерфейса рекомендуется использовать выпадающий список ListBox;
  • Label (нередактируемые) — поля, значение которых задано сервисом и не заполняется пользователем. В качестве компонента пользователького интерфейса рекомендуется использовать Label.

Заполнение формы платежа

PUT /OpenApi/payments/{paymentId}

PUT /OpenApi/payments?externalId={externalId}

  • PaymentId - идентификатор платежа полученный при его формированнии.
  • ExternalId - идентификатор платежа на стороне приложения

{

   "Params":[

      {

         // Идентификатор поля платежа

         // Идентификатор поля платежа

Получение кода подтверждения платежа

POST /OpenApi/payments/{paymentId}/code

В зависимости от настройки, для проведения платежа может быть необходим одноразовый код, который в этом случае запрашивается на последней форме (скалярное поле с идентификатором "$OtpCode"). Указанный запрос инициирует отправку кода подтверждения платежа на мобильный телефон пользователя.

Получение состояния платежа

GET /OpenApi/payments/{paymentId}/state

Пример ответа:

{

   "StateId":"Created"

}

В случае, если платеж не найден, возвращает 404 NotFound.

Возможные состояния ( * отмечены конечные состояния):

  • Created – платеж создан (идентификатор зарезервирован)
  • Updated – платеж изменен (внесены данные одного или нескольких шагов, актуально для многошаговых провайдеров)
  • Blocked – платеж блокирован системой.
  • Processing – платеж в находится в обработке сервером ЕК.
  • Checking – платеж проходит проверку на стороне провайдера.
  • Paying – платеж в обработке на стороне провайдера.
  • ProcessError* – при обработке платежа произошли ошибка.
  • CheckError* – ошибка проверки платежа.
  • PayError* – ошибка проведения платежа на стороне провайдера.
  • Canceled* – платеж отменен.
  • Paid* – платеж проведен успешно.

Проверка данных платежа

POST /OpenApi/payments/{paymentId}/check

Метод проверки данных платежа служит для валидации данных без списания средств с кошелька клиента.

Метод проверки платежа вызывается после получения финальной формы платежа (идентификатор формы - “$Final”). Получене данной формы означает, что все платежные поля заполнены и валидны с точки зрения формата полей.

После вызова метода платеж передается в процессинг для проверки данных на стороне провайдера услуг. Статус платежа устанавливается в Processing на момент передачи данных и в Checking в момент непосредственной проверки.

После завершения проверки платеж может находиться в одном из следующих состояний (* отмечены конечные состояния):

  • CheckError* – ошибка проверки платежа.
  • Checked – платеж успешно проверен, данные валидны.

В случае выявления ошибки данных проведение платежа далее невозможно. Если проверка прошла успешно, то для завершения платежа требуется передать финальную форму методом заполнения формы платежа.

Ответ аналогичен ответу при получении данных платежа.

В случае, если платеж не найден, либо не все обязательные платежные поля были заполнены, возвращает 404 NotFound.

Шаблоны платежей

Создание и редактирования шаблона платежей

Для создания или редактирования шаблона требуется создать и провести платеж на провайдера SaveTemplate.

Подробно о платежах - в соответствующем разделе документации.

Поля провайдера:

  • TemplateId - идентификатор шаблона (необязательный, передавать в случае, если требуется изменить существующий шаблон).
  • TaskId - идентификатор расписания.
  • PaymentId - идентификатор платежа-прототипа. Новые платежи по шаблону будут создаваться на основе этого платежа.
  • Title - название шаблона.

Редактирования расписание

Редактирование расписания функционально заменено создания нового расписания и установкой нового расписания в шаблон.

Доступен также механизм установки расписания на паузу - через проведение платежа на провайдера ChangeTask.

Подробно о платежах - в соответствующем разделе документации.

Поля провайдера:

  • TaskId - идентификатор расписания;
  • IsPaused - поставлено ли расписание на паузу: 1 - пауза, 0 - нет паузы.
