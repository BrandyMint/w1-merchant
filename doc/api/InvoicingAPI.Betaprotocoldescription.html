Приложение 1. Допустимые PaymentType

Принцип работы

Api предполагает 2 сценария использования: создание счетов и их оплата и создание платежей на кошелек.

Сценарий работы со счетами

1. выставление инвойса и создание платежа по нему в состоянии Created

2. передача дополнительных данных для "проведения" транзакции

3. при необходимости выполнение дополнительных действий для авторизации пользователя (например, магазин должен редиректнуть пользователя на страницу банка для ввода СМС-кода )

4. обработка уведомления о успешном погашении инвойса

Сценарий работы с платежами

1. создание платежа на указанные реквизиты в состоянии Created

2. передача дополнительных данных для "проведения" платежа

3. при необходимости выполнение дополнительных действий для авторизации пользователя (например, магазин должен редиректнуть пользователя на страницу банка для ввода СМС-кода )

4. обработка уведомления об платеже либо  вызов метода "получение информации о платеже"

Авторизация

В случае использования данного API магазином необходимо передавать дополнительный заголовок

X-Wallet-UserId с идентификатор магазина в качестве значения

// пример заголовка авторизации для магазина 119973638979

X-Wallet-UserId: 119973638979

В случае использования API регистатором данный заголовок отсутствует.

Формирование цифровой подписи для магазина

Для данного Api использование цифровой подписи является обязательным условиям, магазин не может воспользоваться этим сервисов, если в настройках кассы у него выключена цифровая подпись.

Запрос должен содержать заголовки X-Wallet-Signature и X-Wallet-Timestamp, содержащий дату формирования запроса в часовом поясе UTC+0. Значение заголовка должно иметь формат yyyy-MM-ddTHH:mm:ss.

Также запрос может содержать заголовок X-Wallet-CustomerId, в котором находится уникальный идентификатор пользователя (плательщика) на стороне магазина.

Цифровая подпись запроса формируется путем конкатенации URL, на который отправляется запрос, номера магазина, идентификатора плательщика (если присутствует), значения заголовка X-Wallet-Timestamp, тела запроса и секретного ключа магазина. Полученная строка, представленная в кодировке UTF-8, подписывается с помощью выбранного алгоритма ЭЦП, после чего байтовое представление подписи кодируется в Base64:

Signature = Base64(SignatureMethod(UTF8(

URL + MerchantId + CustomerId + Timestamp + RequestBody + SecretKey)));

На запросы сервер генерирует ответ, который так же должен быть проверен на подлинность, при этом ответу добавляются заголовки:

  • X-Wallet-Timestamp - дата формирования ответа в часовом поясе и формате аналогичном дате формирования запроса.
  • X-Wallet-Signature - ЭЦП ответа сервера.

Цифровая подпись ответа формируется путем конкатенации значения подписи запроса, заголовка ответа X-Wallet-Timestamp, тела ответа и секретного ключа магазина. Полученная строка преобразовывается и хешируется аналогично алгоритму для запросов:

Signature = Base64(SignatureMethod(UTF8(

IncomingSignature + Timestamp + ResponseBody + SecretKey)))

Метод формирования ЭЦП и «секретный ключ» можно настроить в личном кабинете учетной записи магазина на вкладке «Настройки» в разделе «Интернет-магазин»

Формирование цифровой подписи для регистратора

Выполняется аналогично формированию подписи для магазина за исключением следующих пунктов:

  1. подпись формируется с использованием сертификатов
  2. Входящая подпись проверяется по строке
    "    URL + Timestamp + CustomerId + RequestBody"
    а исходящая -
    "    IncomingSignature + Timestamp + ResponseBody"

Создание счета

X-Wallet-Signature: 0J7Qv9C70LDRgtCwINC30LDQutCw0LfQsA==

X-Wallet-Timestamp: 2015-01-07T21:45:33

Передача данных платежного инструмента

Метод необходим для проведения платежа через платежный шлюз. В качестве входных параметров передаются данные для авторизации - номер телефона, данные кредитной карты и пр.

Указанные наборы данных могут отличаться для различных paymentType-ов.

В ответ вернется сущность "Платеж" а также в случае необходимости данные для дополнительной авторизации.

POST /w1/invoicingapi/payments/{paymentId}/process

Content-type: application/json; charset=utf-8

X-Wallet-Signature: 0J7Qv9C70LDRgtCwINC30LDQutCw0LfQsA==

X-Wallet-Timestamp: 2015-01-07T21:45:33

X-Wallet-CustomerId: user108976

{

    "AuthData": { //объект с авторизационными данными

        "CreditCardNumber":"4200000000000000",

Возможные значения AuthType

Redirect

означает, что нужно на HttpForm.Action методом HttpForm.Method отправить данные из HttpForm.Params (key1=value1&key2=value2...)

PayerConfirmation

шлюз ожидает действий от пользователя, например отправку смс.
